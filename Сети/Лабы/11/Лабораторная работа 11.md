  * `Router0` интерфейс к этой сети: **G0/1** → `192.168.3.1/24`

## 1) Сборка топологии / физическое подключение

Подключите интерфейсы так:

* `Router0 F0/0` → к switch → к хостам 192.168.1.0/24
* `Router0 F0/1` ←→ `Router1 F0/1` (свитч или прямое соединение) — сеть 192.168.3.0/24
* `Router1 F0/0` → switch → WWW/DNS сервер (192.168.2.2) и хосты 192.168.2.0/24

Проверьте link-состояния (на маршрутизаторах):

```
Router> enable
Router# show ip interface brief
```

Ожидаем: соответствующие Gigabit интерфейсы «up/up» (если нет — проверьте кабели / переключатели / настройки порта).

## 2) Базовая настройка Router0

На `Router0`:

1. Зайти в конфиг:

```
Router0> enable
Router0# configure terminal
```

2. Настроить hostname (если нужно):

```
Router0(config)# hostname Router0
```

3. Настроить интерфейсы и включить:

```
Router0(config)# interface FastEthernet0/0
Router0(config-if)# ip address 192.168.1.1 255.255.255.0
Router0(config-if)# ip nat inside
Router0(config-if)# no shutdown
Router0(config-if)# exit

Router0(config)# interface FastEthernet0/1
Router0(config-if)# ip address 192.168.3.1 255.255.255.0
Router0(config-if)# ip nat outside
Router0(config-if)# no shutdown
Router0(config-if)# exit
```

4. Сохранить конфиг (позже, после всех проверок — но можно и сейчас):

```
Router0# write memory
```

**Проверки сразу после настройки интерфейсов:**

```
Router0# show ip interface brief
```

Ожидается: IP-адреса назначены, интерфейсы `up`.

---

## 3) Базовая настройка Router1 (пароль telnet и enable secret)

На `Router1`:

1. Войти и задать hostname:

```
Router1> enable
Router1# configure terminal
Router1(config)# hostname Router1
```

2. Настроить интерфейсы:

```
Router1(config)# interface FastEthernet0/0
Router1(config-if)# ip address 192.168.2.1 255.255.255.0
Router1(config-if)# no shutdown
Router1(config-if)# exit

Router1(config)# interface FastEthernet0/1
Router1(config-if)# ip address 192.168.3.2 255.255.255.0
Router1(config-if)# no shutdown
Router1(config-if)# exit
```

3. Установить пароль для перехода в привилегированный режим:

```
Router1(config)# enable secret cisco
```

4. Настроить пароль для доступа по Telnet к линии vty и включить telnet:

```
Router1(config)# line vty 0 4
Router1(config-line)# password cisco
Router1(config-line)# login
Router1(config-line)# transport input telnet
Router1(config-line)# exit
```

(Опционально: `service password-encryption` — чтобы пароли были в конфиге не в открытом виде.)

**Проверки сразу:**

```
Router1# show running-config | section line vty
Router1# show running-config | include enable secret
Router1# show ip interface brief
```

Проверьте, что `line vty` имеет `login` и `password` и `transport input telnet`.

## 4) Ограничить доступ по Telnet к Router1 — Access-list + ip access-class

Требование: по telnet к Router1 можно подключаться **только** с хостов из сети `192.168.2.0/24`.

На `Router1` добавим **стандартный** ACL (подходит для ip access-class на vty):

```
Router1(config)# access-list 1 permit 192.168.2.0 0.0.0.255
```

Потом назначим его на линии vty:

```
Router1(config)# line vty 0 4
Router1(config-line)# access-class 1 in
Router1(config-line)# exit
```

**Сохраняем:**

```
Router1# write memory
```

**Проверки:**

* Просмотр ACL и счетчика совпадений:

```
Router1# show access-lists 1
```

Должны увидеть строку `access-list 1 permit 192.168.2.0 0.0.0.255` и счетчик hits (в дальнейшем при попытках telnet будут увеличиваться).

* Проверка доступа с хоста внутри сети 192.168.2.0/24:

  * С PC(в нашем случае с сервере по настройке сети) (192.168.2.x) выполнить: `telnet 192.168.2.1` — должен запросить пароль (cisco) и успешно войти (после ввода enable secret — при переходе в enable).
* Проверка доступа с хоста вне сети (например 192.168.1.2):

  * `telnet 192.168.3.2` — должен быть отказ (connection refused / no route / access denied). После неудачной попытки `show access-lists 1` должен показать увеличение счетчика deny/permit (для стандартного ACL будет показывать hits).

Примечание: `ip access-class` проверяет **IP-адрес источника** при подключении к vty — именно то, что нам нужно.

## 5) Настроить EIGRP между Router0 и Router1 (AS 10)

Настраиваем на обоих маршрутизаторах:

На Router0:

```
Router0# configure terminal
Router0(config)# router eigrp 10
Router0(config-router)# network 192.168.3.0 0.0.0.255
Router0(config-router)# network 192.168.1.0 0.0.0.255
Router0(config-router)# no auto-summary
Router0(config-router)# exit
```

На Router1:

```
Router1# configure terminal
Router1(config)# router eigrp 10
Router1(config-router)# network 192.168.3.0 0.0.0.255
Router1(config-router)# network 192.168.2.0 0.0.0.255
Router1(config-router)# no auto-summary
Router1(config-router)# exit
```

**Проверки сразу:**

```
Router0# show ip eigrp neighbors
Router1# show ip eigrp neighbors

Router0# show ip route eigrp
Router1# show ip route eigrp

Router0# show ip protocols
Router1# show ip protocols
```

Ожидаем: соседи EIGRP видят друг друга, маршруты EIGRP обменяны (в таблице маршрутизации маршруты 192.168.2.0/24 и 192.168.1.0/24 должны быть видимы у соответствующих роутеров).

## 6) Ограничение транзитного трафика на Router1 со стороны Router0

Задача: **на Router1** ограничить транзитный трафик, приходящий со стороны Router0 (т.е. входящий на интерфейс F0/1). Разрешить только:

* ICMP от любого к любому;
* DNS (UDP порт 53) от любого источника к DNS-серверу (`192.168.2.2`);
* TELNET (TCP порт 23) от любого к любому;
* WWW (HTTP TCP порт 80) **из сети 192.168.3.0/24 к WWW серверу** (`192.168.2.2`);
* трафик протокола EIGRP.

Реализуем расширенный ACL (например `101`) и применим его **входящим** на интерфейс `GigabitEthernet0/1` (интерфейс к Router0).

На Router1:

```
Router1(config)# access-list 101 permit icmp any any
Router1(config)# access-list 101 permit udp any host 192.168.2.2 eq domain
Router1(config)# access-list 101 permit tcp any any eq 23
Router1(config)# access-list 101 permit tcp 192.168.3.0 0.0.0.255 host 192.168.2.2 eq www
Router1(config)# access-list 101 permit eigrp any any
Router1(config)# access-list 101 deny ip any any
```

(последняя строка — явно запрещаем всё остальное; не обязательно, т.к. implicit `deny ip any any` уже есть, но лучше прописать для наглядности).

Применяем на интерфейсе, со стороны Router0:

```
Router1(config)# interface FastEthernet0/1
Router1(config-if)# ip access-group 101 in
Router1(config-if)# exit
```

**Проверки сразу:**

* Проверить что ACL применён:

```
Router1# show ip interface FastEthernet0/1
```

В выводе увидите `Inbound access list is 101`.

* Посмотреть содержимое и счётчики ACL:

```
Router1# show access-lists 101
```

Пригенерации трафика здесь вы увидите увеличение счетчиков hits.

* Проверить, что EIGRP-соседство не разрушилось (ACL разрешает eigrp):

```
Router1# show ip eigrp neighbors
Router0# show ip eigrp neighbors
```

* Проверка примерных сценариев (с PC/серверов):

  * **ICMP**: с `192.168.1.2` → `ping 192.168.2.2` — должен проходить (ICMP разрешён).
  * **Телнет**: с любого хоста → `telnet 192.168.2.2 23` — разрешён (ACL разрешает tcp any any eq 23).
    *Замечание:* Telnet к серверу — если на сервере слушается telnet.
  * **DNS**: от любого хоста → DNS (udp 53) к `192.168.2.2` — должен проходить.
  * **WWW**: важно понимать сценарий: хосты из 192.168.3.0/24 (включая адрес интерфейса Router0 192.168.3.1, используемый для NAT) → `tcp dst 192.168.2.2 eq 80` — разрешён.
    Попробуйте: с `192.168.3.3` → `telnet 192.168.2.2 80` или `curl http://192.168.2.2` — трафик 80 должен пройти.
  * **Прочий трафик**: например FTP/SMTP/др. — должен блокироваться (deny ip any any).

Если какой-то нужный трафик блокируется — проверяйте `show access-lists 101` чтобы увидеть какие строки считаются совпадениями.

## 7) Настройка PAT (NAT overload) на Router0

Требование: хосты из сети `192.168.1.0/24` должны обращаться к WWW серверу **от имени** адреса интерфейса Router0 в сети `192.168.3.0/24` (т.е. source NAT/PAT к 192.168.3.1).

На Router0:

1. Создаём стандартный access-list, который определяет, какие адреса будут NAT'иться:

```
Router0(config)# access-list 10 permit 192.168.1.0 0.0.0.255
```

2. Включаем PAT, используя интерфейс F0/1 (тот что 192.168.3.1):

```
Router0(config)# ip nat inside source list 10 interface FastEthernet0/1 overload
```

3. Убедитесь, что на интерфейсах проставлены `ip nat inside` / `ip nat outside` (мы делали это раньше):

* `F0/0` (192.168.1.1) → `ip nat inside`
* `F0/1` (192.168.3.1) → `ip nat outside`

(Если вы забыли — добавьте команды как в шаге 2).

**Проверки NAT сразу:**

* Посмотреть NAT-таблицу:

```
Router0# show ip nat translations
```

Сначала таблица может быть пустой до тех пор, пока не будет активного NAT-сеанса.

* Посмотреть статистику NAT:

```
Router0# show ip nat statistics
```

* Провести тест: с хоста `192.168.1.2` выполните в браузере `http://192.168.2.2` (или `ping 192.168.2.2` и/или `telnet 192.168.2.2 80`), трафик должен уходить через Router0 → Router1 → сервер. На Router0 после установленного соединения в `show ip nat translations` появится запись, например:

```
Pro  Inside global         Inside local          Outside local         Outside global
tcp  192.168.3.1:xxxx      192.168.1.2:yyyy      192.168.2.2:80        192.168.2.2:80
```

(здесь xxxx/yyyy — порты).

* Дополнительно:

```
Router0# show running-config | include ip nat
Router0# show running-config | section access-list
```

Если `show ip nat translations` пуст — значит трафик не проходит до Router0 или запросы не доходят до внешнего интерфейса. В этом случае:

* Проверьте, что хости имеют корректный gateway (192.168.1.1).
* Проверьте маршрут до 192.168.2.0/24 (через EIGRP маршруты должны быть).

## 8) Проверки попадания пакетов в ACL’ы и в NAT

### Проверка ACL (Router1):

* `show access-lists 1` — проверки для vty (telnet) — смотрите счетчики hits при попытках входа.
* `show access-lists 101` — проверьте количество hits для каждого permit/deny после тестов.
* `show ip interface GigabitEthernet0/1` — убедиться, что ACL 101 висит inbound.

### Проверка NAT (Router0):

* `show ip nat translations` — должно быть по крайней мере одно вхождение после тестового HTTP/HTTPS-запроса.
* `show ip nat statistics` — общее число трансляций / адреса интерфейса.
* `clear ip nat translation *` — почистит таблицу (используйте для повторных тестов).
* При желании используйте `debug ip nat` (будьте аккуратны в лабораторной сети — вывод может быть большой):

```
Router0# debug ip nat
```

и потом `undebug all` чтобы выключить.

## 1️⃣ Использовать **Router** или **PC** как Telnet-сервер

### Вариант A: Router как Telnet-сервер

1. Кликаешь по роутеру → CLI → включаем конфигурацию:

```cisco
enable
configure terminal
line vty 0 4
password cisco      ! задаём пароль
login
exit
enable secret cisco  ! необязательно, но для безопасности
```

2. Убедись, что интерфейсы роутера подняты (`no shutdown`):

```cisco
interface g0/0
ip address 192.168.2.1 255.255.255.0
no shutdown
exit
```

3. Теперь с сервера в сети можно тестировать Telnet:

```
telnet 192.168.2.1
```

✅ Подключение должно пройти, запросит пароль.

---

### Вариант B: PC → Telnet клиент

На **PC → Desktop → Command Prompt** есть встроенная команда `telnet` (клиент), но **PC сам сервер Telnet не поддерживает**.

То есть **Server в Packet Tracer** не даст прям Telnet. Для тестов используем **Router** как сервер, а PC как клиент.
