# 2) Настройка RIP (всем маршрутизаторам)

Требования: информация обо всех сегментах сети должна присутствовать у каждого маршрутизатора; используются VLSM; RIPv2 (multicast) — включим `version 2` и `no auto-summary`. R2 не должен посылать RIP-апдейты через интерфейсы Lo0 и Fa0/0 (подключённый к коммутатору) — реализуем `passive-interface` на R2 для этих интерфейсов.

> Замечание: в IOS команда `network` в RIP принимает classful адрес — используем `network 10.0.0.0`, `network 192.168.1.0`, `network 192.168.2.0`. Это корректно и включит нужные интерфейсы (RIPv2 + no auto-summary позволяет VLSM).

**R0 — RIP:**

```
configure terminal
router rip
 version 2
 no auto-summary
 network 10.0.0.0
 network 192.168.1.0
 network 192.168.2.0
exit
end
write memory
```

**R1 — RIP:**

```
configure terminal
router rip
 version 2
 no auto-summary
 network 10.0.0.0
 network 192.168.2.0
exit
end
write memory
```

**R2 — RIP (и passive-interface для Lo0 и Fa0/0):**

```
configure terminal
router rip
 version 2
 no auto-summary
 network 10.0.0.0
 network 192.168.1.0
 network 192.168.2.0
 passive-interface FastEthernet0/0
 passive-interface Loopback0
exit
end
write memory
```

Проверки (после конфигурации RIP выполнить на каждом маршрутизаторе):

**На всех маршрутизаторах:**

```
show ip protocols      <-- покажет, какие сети включены в RIP, версию, таймеры и passive interfaces
show ip route | include ^R   <-- покажет маршруты, полученные через RIP (R)
show ip route          <-- полная таблица, проверьте, что сети всех трёх LAN и линков видны
```

# 4) Настройка EIGRP (всем маршрутизаторам)

Требование: **не включать в апдейты EIGRP информацию об IP сетях интерфейсов Lo0** маршрутизаторов. Отключаем автоматическую суммаризацию `no auto-summary`. Для этого в network используем точные wildcard-маски (инвертированные маски), чтобы покрыть только физические сети и линковые /30 и LAN /25, а не loopback /32.

Выберем autonomous system = `100` (пример).

**R0 — EIGRP:**

```
configure terminal
router eigrp 100
 no auto-summary
 network 192.168.1.0 0.0.0.127
 network 192.168.2.248 0.0.0.3
exit
end
write memory
```

**R1 — EIGRP:**

```
configure terminal
router eigrp 100
 no auto-summary
 network 192.168.2.0 0.0.0.127
 network 192.168.2.248 0.0.0.3
 network 192.168.2.252 0.0.0.3
exit
end
write memory
```

**R2 — EIGRP:**

```
configure terminal
router eigrp 100
 no auto-summary
 network 192.168.1.128 0.0.0.127
 network 192.168.2.252 0.0.0.3
exit
end
write memory
```

Проверки EIGRP (на каждом маршрутизаторе):

```
show ip protocols      <-- увидите EIGRP process, сети и no auto-summary
show ip eigrp neighbors
show ip eigrp topology
show ip route | include ^D    <-- маршруты EIGRP (маркер D)
show ip route
```

**Важно:** мы специально не включили в EIGRP сети Lo0 (10.x.x.x), поэтому loopback-сети будут НЕ распространяться через EIGRP.

# 5) Анализ после RIP+EIGRP (пункт 6 лабораторной) — какие протоколы доставляют маршруты и почему

После одновременной работы RIP и EIGRP вы увидите в `show ip route`:

* Для сетей LAN и линковых /30 (которые объявлены и в EIGRP и в RIP) — маршруты предпочтёт **EIGRP**, потому что **административная дистанция (AD)** EIGRP по умолчанию = **90**, а RIP = **120**. Поэтому в таблице маршрутов будут стоять метки `D` (EIGRP).
* Для **loopback-сетей 10.1.1.1/32, 10.2.2.2/32, 10.3.3.3/32**: мы исключили их из EIGRP, но включили в RIP (network 10.0.0.0 в RIP), поэтому эти loopback-сети будут распространяться через **RIP** и в таблицах будут помечены `R`.

Проверки:

```
show ip route
show ip route | include ^D
show ip route | include ^R
```

Таким образом: EIGRP обслуживает (предпочтительно) основные «физические» сети, RIP — loopback-адреса (пока OSPF не включён).

# 6) Настройка OSPF (всем маршрутизаторам) — все интерфейсы в Area 0 (пункт 7)

# 7) Анализ после включения OSPF (пункт 8) — какие протоколы используются для достижения сетей

По умолчанию AD: EIGRP=90, OSPF=110, RIP=120.

* Для сетей, у которых есть сведения и в **EIGRP** и в **OSPF** — по-умолчанию **EIGRP** будет иметь приоритет (AD 90 < 110), поэтому маршруты останутся помеченными `D`.
* Для сетей, которые **не были включены в EIGRP** (например Lo0, которые мы исключили из EIGRP) — эти сети будут распространяться через **OSPF** (если мы включили Lo0 в OSPF), или через RIP (если OSPF не включает их). В нашей конфигурации loopback объявлены в OSPF (мы включили `network 10.x.x.x` в OSPF), поэтому:

  * **Loopback-сети** будут теперь распространяться **OSPF** (потому что мы добавили их в OSPF). R1/R2/R0 получат эти сети как OSPF (`O`), если EIGRP не содержит их.
* Итого: после включения OSPF большинство loopback-сетей будут `O`, физические сети останутся `D` (EIGRP) — если не предпринять специальных мер.

Проверка (примерно на R0/R1):

```
show ip route
show ip route | include ^D
show ip route | include ^O
```

# 8) Сделать так, чтобы R1 использовал только OSPF (пункт 9)

Требуется: **R1 должен использовать только информацию OSPF** для достижения всех IP сетей. Самый прямой путь — на R1 изменить AD для RIP и EIGRP так, чтобы они стали хуже, чем OSPF (т.е. >110). Тогда OSPF (110) будет иметь приоритет.

На R1:

**Поднять AD для RIP и EIGRP на R1 (пример: 200):**

```
configure terminal
router ospf 1
 distance 70
exit
end
write memory
```

(Команда `distance <x>` внутри конфигурации конкретного routing process задаёт административную дистанцию для маршрутов, изученных этим процессом, на данном роутере.)

Проверки на R1:

```
show ip protocols           <-- посмотрите, что distance изменился для соответствующих процессов
show ip route               <-- теперь большинство маршрутов должны быть помечены O (OSPF)
show ip route | include ^O
show ip route | include ^D
show ip route | include ^R
```

Ожидаемый результат: в таблице маршрутизации R1 маршруты, которые доступны через OSPF, будут ставиться O (и использоваться), а маршруты, которые приходят через EIGRP/RIP, будут либо оставаться, но помечены с более высокой AD и не будут устанавливаться как активные записи (или будут помечены и меткой другого протокола). В результате R1 использует информацию OSPF для всех сетей.

# 10) DR и BDR для сегментов сети (пункт 11) — кто и почему

Правила OSPF DR/BDR (кратко):

* Выбор проходит только на многодоступных сетях (broadcast) — т.е. Ethernet-сегменты за L2-коммутаторами.
* На point-to-point (например /30 между двумя роутерами) DR/BDR **не выбираются**.
* Устройство с наивысшим `priority` (default 1) выигрывает; при равных priority — выбирается наивысший Router ID.

В нашей топологии:

* Сегмент `192.168.1.0/25` (между R0 и свичем) — в OSPF участвует только **R0** (на этой сети нет других OSPF-маршрутизаторов). R0 будет считаться DR на этом сегменте (нет конкуренции). BDR отсутствует.
* Сегмент `192.168.2.0/25` (LAN R1) — участвует только **R1** → R1 будет DR.
* Сегмент `192.168.1.128/25` (LAN R2) — участвует только **R2** → R2 будет DR.
* Линки R0–R1 и R1–R2 — это /30 point-to-point: **DR/BDR не применяются**.

Проверка DR/BDR и приоритетов:

```
show ip ospf interface FastEthernet0/0    <-- на соответствующем роутере
```

Вывод покажет строки:

* `Designated Router (ID) ...`
* `Backup Designated Router (ID) ...`
  Если на сегменте только один роутер — он будет DR, BDR = `0.0.0.0` (или отсутствует).

Также:

```
show ip ospf neighbor    <-- покажет состояния соседей, а на multiaccess покажет DR и BDR
```

Обоснование: потому что на каждом LAN только один OSPF-роутер, он автоматически становится DR; на point-to-point DR/BDR не используются.
